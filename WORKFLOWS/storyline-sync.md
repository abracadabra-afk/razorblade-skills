# WORKFLOW: StoryLine Sync (mirror the chapter into StoryLine)

> Derivation pass that converts a **landed** `draft.md` into StoryLine-format files — one `.md` per scene plus Codex entries for the characters, locations, and world it touches — so CRE gets StoryLine's analytical views (Board, Timeline, Plotlines subway map, Relationship Map, Stats) over prose he wrote in the per-chapter pipeline. StoryLine stores everything as Markdown + YAML frontmatter, so this is a pure file-derivation pass, no API. Established by the Chapter 1 (Knots) pilot, 2026-06-09 (see [[_OBSERVATIONS]]).
>
> **`draft.md` is the single source of truth. StoryLine is a regenerated mirror — never authored in directly.** This is the load-bearing rule: if CRE drafts prose inside StoryLine, the two systems drift. StoryLine is downstream; this skill keeps it downstream.

## When to use

After a chapter's `draft.md` has landed (post-[[WORKFLOWS/promote-revision]]), ideally right after [[WORKFLOWS/canon-sync]] so the bible/threads facts this pass reads are current. Trigger phrases: "sync to storyline," "build the storyline," "storyline this chapter." Do NOT use it to revise prose ([[WORKFLOWS/register-pass]]), update REFERENCE canon ([[WORKFLOWS/canon-sync]]), or author StoryLine content by hand. It derives StoryLine files from finished prose; it never changes a word of any `draft.md`.

## Key principles

1. **Derive only from landed text + REFERENCE.** Scene prose comes from `<chapter>/draft.md`; codex facts come from `REFERENCE/bible.md`; plotline tags come from `REFERENCE/threads.md`; scene end-state (for scene `notes`) from `<chapter>/continuity.md`. Never derive from `slate/` or `revisions/`. Scaffold or pre-register `status` → warn and ask before syncing.
2. **One direction only.** Prose flows `draft.md` → StoryLine, never back. StoryLine is the lens; the pipeline is the workshop.
3. **Re-sync is a merge, not a clobber.** StoryLine writes its own frontmatter fields back into scene files (`wordcount`, `modified`, `notesFile`, corkboard position) and CRE hand-tunes others (`intensity`, `emotion`, `status`, `tags`). On re-sync: **regenerate the prose body** from `draft.md`, but **preserve** (a) StoryLine-owned fields and (b) any field CRE has changed from this skill's last derived value. Only refresh derived-from-prose facts (characters present, location, conflict). When in doubt, tag `<<UNCERTAIN: …; confirm?>>` rather than overwrite — same discipline as canon-sync.
4. **Inferred intensity = three flagged axes, never asserted.** A scene carries all three (CRE ruling 2026-06-09): native `intensity` = **`round(si_local/10)`**, the engine's 0–10 pacing/menace scalar — on the native field because StoryLine's tension curve is hardwired to it; custom `arc_valence` = signed **−10…+10** good/bad direction; the scene-intensity engine's `si_local`/`si_band`/`si_dread`/`si_scores` = the reproducible full read. Plus `emotion` (subjective). Always record in scene `notes` as "intensity/emotion inferred — adjust to taste," and note the `si_*` engine version. They are a first pass for CRE to correct, not ground truth. `intensity` is engine-derived, but its absolute value is still subject to the D5 recal (`^scene-intensity-d5-recal`). **The `si_*` values MUST be produced by actually running the scene-intensity engine (Step 3b) — never hand-estimated. Eyeballing `si_local` silently reintroduces the run-to-run variance the engine exists to remove, which is the whole reason it exists.**
5. **Threads become plotline tags — all touched.** Each `threads.md` entry the scene plants, advances, OR pays off becomes a scene `tag` (kebab-case of the thread name) — bare plantings included (CRE ruling 2026-06-09). **The tags are the whole job:** StoryLine auto-derives the Plotlines subway map from scene tags and auto-colors each line by name-hash — no registry, no color file (see Step 5b). Get the tags right and the map renders distinctly on its own.
6. **Codex is idempotent and spoiler-clean.** Codex entries derive from bible facts carrying their `(CH<N> rev<M>)` provenance; a character/location already present is updated in place, not duplicated. Respect the bible's reveal-point discipline — do not write a later-chapter fact into an early-chapter codex entry.

## Steps

### Step 0 — Vault sentinel
Read `_DIRECTIVES.md`; confirm frontmatter `type: ai-os-brain` + `file: directives`. Mismatch or missing → halt and ask which folder is the vault. (Shared `^obs-004` gate.)

### Step 1 — Locate chapter + target project; verify landed
Resolve the chapter folder and read `draft.md` frontmatter — expect `status: register-revised` (or `dev-revised` if CRE syncs mid-pipeline). Record `source_revision` as the provenance tag. Resolve the target StoryLine project folder under `WRITING/STORYLINE/<Project>/` (confirm with CRE if more than one exists). Scaffold/missing draft → halt.

### Step 2 — Load sources + current StoryLine state
Read `REFERENCE/bible.md`, `REFERENCE/threads.md`, the chapter's `continuity.md`, and any existing files in `WRITING/STORYLINE/<Project>/Scenes/` and `/Codex/`. Index existing scenes by title and existing codex entries by name, capturing StoryLine-owned and CRE-tuned frontmatter values for the merge in Step 5.

### Step 3 — Segment into scenes
Split the chapter into StoryLine scenes. **Ruled rule (CRE 2026-06-09): one scene per `***` (or other hard scene-break) in the draft**; a chapter with no break is one scene. No word-count floor/cap. Cross-check against the chapter's envelope segments (e.g. `waking-hearth` / `the-hunt`) as a sanity check on the count, not as the split unit. Assign `sequence` in reading order across the whole project (continue from the highest existing sequence; renumber only with CRE's OK). Map `act`/`chapter` from the draft's chapter frontmatter as **plain integers — no beat sheet** (ruled call #5; do not write a StoryLine `beatsheet` or apply a stock scaffold).

### Step 3b — Score scene intensity (RUN the engine; do not estimate)
The intensity axes Step 4 writes are **not hand-derived** — they are the output of actually running the scene-intensity engine on the scenes segmented in Step 3. scene-intensity is its own skill/engine; this sync is its *caller*. Per segmented scene:

1. **Judgment dims (model).** Score D1 (stakes), D2 (proximity), D3 (turns), D4 (irreversibility) with quoted evidence, and apply the carried-dread gate (most scenes 0; cite a `dread_source` for any +1/+2; set `on_page_detonation: true` where the established threat acts in full view). Honor the project rulings: the **D2 predator-POV clause** (a hunt of unbound prey does not raise D2) and the **juxtaposition-dread hand-flag** (an off-page-catastrophe scene scores low locally but is flagged reader-HIGH by hand — the engine's documented blind spot).
2. **Build `scenes.json`** with those dims + each scene's `prose`.
3. **Run the engine:** `python scene_intensity.py score scenes.json --config intensity_config.json`. It computes D5/D6 mechanically (dialogue-aware), the weighted composite, the band, the spread-based seam, and the carried-dread band math — deterministically (same input → same output).
4. **Carry the output into Step 4 verbatim:** native `intensity = round(si_local/10)`; the `si_local`/`si_band`/`si_seam`/`si_dread`/`si_reader_band`/`si_scores` block as the engine returned them; `arc_valence` (signed direction) by judgment. Record all as inferred in `notes`, with the engine version.

Skipping this step (writing `si_*`/`intensity` by eye) is the one move that silently breaks the instrument — see principle 4. Use the patched engine (`_skill-patches/scene-intensity/`, or the installed skill once rebundled), not the stock horror-tuned defaults.

### Step 4 — Extract per scene + build codex
For each scene, derive the frontmatter from the prose + sources (see Schema reference): `pov`, `characters` (as `[[wikilinks]]`), `location` (as `[[wikilink]]`), `storyDate`/`storyTime`, `conflict` (one sentence), `emotion` + the three intensity axes from Step 3b's engine output (`intensity` = `round(si_local/10)` + `arc_valence` + the `si_*` block, all flagged inferred), `tags` (all touched threads), `status` (via the full lifecycle map: `scaffold→idea`, slate→`draft`, `dev-revised→written`, `register-revised→revised`, `final→final`), `timeline_mode` if non-linear, and a `notes` block recording source, inferred-field caveat, threads touched, and end-state from `continuity.md`. For every character/location/world/**object/lore-rule** a scene references, create or update its Codex entry from `bible.md`, including structured `relations`. **Mirror character arc state from `arcs.md` into the Codex character fields StoryLine already has (no new fields):** entry state → `startingPoint`; exit state (current) → `goal`/`expectedChange`; the character's key relationship arcs → `relations[]`; and the waypoint trajectory + arc_type into the codex `notes` body, flagged inferred (`arc inferred from arcs.md — adjust to taste`), same caveat discipline as the intensity fields. Spoiler discipline (principle 6) holds: write only state through the chapter being synced — do not pull a later chapter's exit state into an earlier scene's codex entry. Mirror the bible's sections to Codex categories — **all are built-in StoryLine categories**: Characters→`Codex/Characters/`, Places→`Codex/Locations/` (both always-on), **Objects→`Codex/Items/`** and **Lore & rules→`Codex/Lore/`** (ruled call #6). `items` and `lore` are *toggleable* built-ins (alongside `creatures`, `organizations`, `culture`, `systems`) governed by the **global** plugin setting `codexEnabledCategories` in `data.json` — NOT per-project, and NOT custom categories (`codexCustomCategories` stays empty). The sync does **not** write that setting (global, app-owned, vault-synced — see Step 6 + `^obs-017`); it assumes the needed categories are enabled and verifies it. Then populate: Petyr the Brave, Remedy, the hunter's gear → `Codex/Items/`; the lore rules → `Codex/Lore/`. Use the project's existing names exactly so wikilinks resolve.

### Step 5 — Gate conflicts, then write (merge)
If a derived value contradicts existing StoryLine data that looks author-tuned (CRE moved a scene to a different act; changed a name), stop and present the conflict (existing value + new evidence + recommendation) for CRE's ruling. Otherwise write:
- `Scenes/<NN> - <Title>.md` — regenerate the prose body from `draft.md`; merge frontmatter per principle 3 (preserve StoryLine-owned + CRE-tuned fields, refresh derived facts).
- `Codex/Characters/<Name>.md` and `Codex/Locations/<Name>.md` (+ the world file) — add/update entries idempotently, gaps-only on author-written lines.
Omit any field with no data (StoryLine dislikes empty strings/lists). Filenames: `<sequence> - <title>.md` for scenes, `<Entity Name>.md` for codex.

> **YAML emission discipline (MANDATORY — `^obs-025`/`^obs-028`/`^obs-029`, recurred 3×; DIR-004).** Do **not** hand-format frontmatter. Use the bundled serializer the skill now ships — `scripts/write_storyline.py --out "<PATH>" --frontmatter @fm.json --body @body.md` (or `--body-stdin` for large prose) — which builds the file from a dict via `yaml.safe_dump(d, default_flow_style=False, sort_keys=False, allow_unicode=True, width=100000)`, wraps it in `---\n…---\n\n`, writes the prose body **verbatim**, and **gates the write on a real `yaml.safe_load` round-trip** of what it just wrote (exit 1 on a parse/round-trip failure). The equivalent by hand: build a Python dict and emit it with `yaml.safe_dump(d, default_flow_style=False, sort_keys=False, allow_unicode=True, width=100000)`, then wrap in `---\n…---\n\n` and append the prose body. safe_dump auto-escapes every value that has bitten this lane: a wikilink with an apostrophe (`[[Baby Bird's Box]]`), a `significance`/`notes`/`conflict` value that starts with `"` or contains `: `, a list of `[[wikilinks]]`. If you ever must hand-write a single-quoted scalar, **double every internal apostrophe** (`'[[Baby Bird''s Box]]'`) and **never start a scalar with a bare `"`**. The Step 6 self-test (below) MUST parse every written file with a real YAML loader via the **file tools / a fresh handle, not the bash mount** (`^obs-014`) — a hand-written file that "looks right" is not verified until a loader accepts it.

### Step 5b — Plotlines: nothing to write (auto-managed)
**The sync writes nothing to `System/` for plotlines.** Verified against the installed plugin (`main.js`, 2026-06-09 reload + screenshot): StoryLine's Plotlines view **auto-derives lines from scene `tags`** (`getAllTags()`), and **auto-colors each line from a hash of the tag name** (`lf()` → `T8()` base-hue hash + global hue/sat/light offsets). So tagging the scenes correctly in Step 4 is the *entire* job — the subway map renders, distinctly colored, with no registry and no color file. The earlier "dark map / monochrome / write `plotlines.json`" theory was wrong twice over: the plugin overwrites `System/plotlines.json` from its own settings on load, and colors are computed on the fly (no color data persists in `data.json` either).
- `tagColors` is an **override only** — used when CRE wants a specific color instead of the auto-hashed one. It is set **in-app** (right-click tag → color) and lives in the plugin's `data.json`. The sync never writes it.
- Do not write `System/plotlines.json` or hand-edit `data.json`; both are plugin-owned. The only plotline lever the sync touches is the scene `tags` themselves (Step 4).

### Step 5c — Plot Grid: nothing to write (native "Sync from Scenes")
**The sync writes nothing to `System/plotgrid.json`.** StoryLine has a native **"Sync from Scenes"** button (Plotgrid view toolbar) that auto-builds the grid — verified by a real run (CRE, 2026-06-09): rows = scenes, columns = characters, and it is **additive + author-safe** (it preserved a hand-made row/column and tagged everything it generated with `sourceType: "auto"` + `sourceId`). CRE clicks the button to refresh; the workflow does not touch the file. Rationale: (a) the plugin self-manages this like Plotlines and Codex categories; (b) a thread×scene grid would duplicate the Plotlines subway map, and the native grid's scene×character cut is complementary, not something the sync should overwrite. *(CRE ruling 2026-06-09. Schema captured below in case a future custom grid is ever wanted — but the default is hands-off.)*
### Step 6 — Verify
Programmatically parse every written file via `scripts/write_storyline.py --verify <files…>` (a real `yaml.safe_load` on a fresh handle, **not** the bash mount — `^obs-014`) and confirm: all frontmatter loads, every scene `characters`/`location` wikilink resolves to a real codex file, `sequence` numbers are unique, `type` is correct (`scene`/`character`/`location`/`world`), and **every scene's `tags` resolve to known `threads.md` entries** (plotlines + their colors are auto-managed by StoryLine from these tags — no color/registry file to check). **Codex-category readiness (read-only):** for any Codex category this sync wrote into beyond the always-on Characters/Locations (e.g. `Items`, `Lore`), confirm it is listed in `data.json codexEnabledCategories`; if a category is missing, **warn** ("entries written to `Codex/Items/` won't display until you enable Items in StoryLine → Codex settings") — do NOT write `data.json` to fix it (global, app-owned). **Intensity-engine check:** confirm every scene's `intensity` equals `round(si_local/10)` and the `si_*` block is present and well-formed — a mismatch or a missing block means Step 3b was skipped or the values were eyeballed; halt and re-run the engine. Report a pass/fail table. (The pilot used a PyYAML check — keep that step; StoryLine silently ignores a malformed file rather than erroring.)

### Step 7 — Log
Append to the chapter `changelog.md` and vault [[_CHANGELOG]] (fiction lane): chapter synced to which StoryLine project, scenes written, codex entries added/updated, conflicts ruled/deferred. File fragilities to [[_OBSERVATIONS]].

## Schema reference (StoryLine YAML)

Source: the plugin's own conversion templates (`StoryLine-Scene-Conversion-Template.md`, `StoryLine-Conversion-Template.md`). Use exact field names; camelCase where shown; omit empty fields; list syntax for arrays.

**Scene** — `type: scene` (req), `title` (req), `act`, `chapter`, `sequence` (reading order), `chronologicalOrder` (only if non-linear), `pov`, `characters` (`[[wikilinks]]`), `location` (`[[wikilink]]`), `storyDate`, `storyTime`, `status` (`idea`→`outlined`→`draft`→`written`→`revised`→`final`; **pipeline→StoryLine map:** `scaffold→idea`, slate→`draft`, `dev-revised→written`, `register-revised→revised`, `final→final`), `conflict`, `emotion`, `intensity` (**`round(si_local/10)`**, the engine pacing/menace scalar — native field drives StoryLine's tension curve), `arc_valence` (**custom**, signed −10…+10 direction), `si_local`/`si_band`/`si_seam`/`si_dread`/`si_reader_band`/`si_scores` (**custom**, scene-intensity engine, full read), `target_wordcount`, `tags` (all touched threads), `setup_scenes`/`payoff_scenes` (file paths), `timeline_mode` + `timeline_strand` (non-linear), `notes` (editorial, frontmatter not body). Body after `---` = manuscript prose. StoryLine writes back `wordcount`, `modified`, `notesFile` — treat as StoryLine-owned. *(Note: StoryLine's stock `intensity` is documented as −10…+10, but this project binds native `intensity` to the 0–10 pacing scalar `round(si_local/10)` and carries direction in custom `arc_valence` — CRE rulings 2026-06-09, "menace-native" + open-call #3.)*

**Character** — `type: character` (req), `name` (req), `tagline`, `nickname`, `age`, `role`, `occupation`, `residency`, `locations[]`, `family`, `relations[]` (`{category, type, target}`), `appearance`, `distinguishingFeatures`, `style`, `quirks`, `personality`, `internalMotivation`, `externalMotivation`, `strengths`, `flaws`, `fears`, `belief`, `misbelief`, `formativeMemories`, `accomplishments`, `secrets`, `startingPoint`, `goal`, `expectedChange`, `habits`, `props`, `custom{}`. Body = `notes`. **Arc mirror (added 2026-06-13, `^obs-048`):** `arcs.md` entry state → `startingPoint`, exit/current state → `goal`/`expectedChange`, relationship arcs → `relations[]`, waypoints + arc_type → `notes` (flagged inferred). No custom fields needed — these are StoryLine's own character-arc fields.

**Location** — `type: location` (req), `name` (req), `locationType`, `world`, `parent` (nesting), `description`, `atmosphere`, `significance`, `inhabitants`, `connectedLocations`, `mapNotes`, `custom{}`. Body = `notes`.
**World** — `type: world` (req), `name` (req), `description`, `geography`, `culture`, `politics`, `magicTechnology`, `beliefs`, `economy`, `history`, `custom{}`. Body = `notes`.
**Item** — `type: item` (req), `name` (req), `itemType`, `description`, `origin`, `history`, `owner` (`[[wikilink]]`), `previousOwners`, `properties`, `limitations`, `significance`. Folder `Codex/Items/` (built-in category `items`). Body = `notes`.
**Lore** — `type: lore` (req), `name` (req), `loreType`, `description`, `fullText`, `sources`, `significance`, `relatedEntries` (`[[wikilinks]]`). Folder `Codex/Lore/` (built-in category `lore`). Body = `notes`. *(Both verified against the plugin's `fieldKeys`, 2026-06-09.)*

**Plot Grid** (`System/plotgrid.json`, **plugin-owned — the sync does NOT write this**; schema captured for reference only, from a real native "Sync from Scenes" run). `{ rows:[…], columns:[…], cells:{…}, zoom, stickyHeaders }`. **Row** `{ id:"r-<base36time>-<rand>", label, height:80, bgColor:"", sourceType?:"auto", sourceId?:"<scene path>" }`. **Column** `{ id:"c-…", label, width:160, bgColor:"", sourceType?:"auto", sourceId?:"<name>", sourceKind?:"characters" }`. **Cell** keyed `"<rowId>-<colId>"` → `{ id, content, bgColor, textColor, bold, italic, align:"center", manualContent?:true, linkedSceneId?:"<scene path>" }`. Native sync is additive: hand-made rows/cols (no `sourceType`) are preserved; generated ones carry `sourceType:"auto"`.

**Relation taxonomy** — `family`: sibling, parent, child, guardian, ward, … · `romantic`: partner, spouse, ex-partner · `social`: ally, friend, best-friend, confidant, acquaintance · `conflict`: enemy, rival, betrayer, avenger · `guidance`: mentor, mentee, leader, follower, commander, … · `professional`: colleague, business-partner, client, handler, asset · `story`: protector, dependent, owes-debt-to, sworn-to, idolizes, fears, obsessed-with · `custom`: any string.

**Project layout** — `WRITING/STORYLINE/<Project>/` → `Scenes/`, `Notes/`, `Archive/`, `Research/`, `Codex/{Characters,Locations,Items,Lore}/` (all **built-in** categories — ruled call #6; `Items`/`Lore` are toggleable via the global `data.json codexEnabledCategories`, no custom registration), `System/` (JSON, plugin-owned — the sync writes NONE of it; plotlines auto-manage, per Step 5b), `Exports/`.
